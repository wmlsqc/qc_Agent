# 智扫通机器人智能客服 Agent 项目汇报说明文档

## 1. 项目背景

本项目是一个面向扫地机器人 / 扫拖一体机器人的智能客服 Agent。它不是一个普通聊天机器人，而是一个可以结合知识库、用户数据、设备状态、耗材状态、清扫历史和故障诊断工具来回答问题的业务型智能体。

传统客服系统通常只能回答固定 FAQ，遇到“我的机器人电量多少”“耗材要不要换”“最近为什么漏扫”“本月使用情况怎么样”这类问题时，需要连接业务数据或人工排查。本项目的目标就是模拟一个更完整的智能客服场景：让 Agent 能够理解用户问题，选择合适工具，读取本地模拟业务数据，再给出结构化回答。

当前项目保持轻量化，没有引入数据库。业务数据使用 CSV，记忆使用 JSON，Web 页面使用 Streamlit，接口扩展使用 FastAPI。

## 2. 应用场景

项目适合演示以下场景：

- 扫地机器人售前咨询：例如产品选购、使用建议、维护保养。
- 售后客服支持：例如无法回充、不出水、异响、漏扫等问题的初步诊断。
- 用户设备状态查询：例如设备是否在线、电量、当前模式、异常状态。
- 耗材保养提醒：例如主刷、边刷、滤网、拖布、集尘袋是否需要更换。
- 清扫效果分析：例如本月清扫次数、覆盖率、漏扫情况和优化建议。
- 系统集成接口：通过 FastAPI 给外部业务系统提供查询能力。

## 3. 技术架构

项目可以理解为五层结构：

```text
用户入口层
  - Streamlit Web 页面
  - FastAPI 接口

Agent 编排层
  - ReactAgent
  - main_prompt 主提示词
  - 轻量规划规则

工具能力层
  - 设备状态查询
  - 耗材状态查询
  - 清扫历史查询
  - 用户画像查询
  - 故障诊断
  - Agent 自检
  - RAG 知识库问答

数据与知识层
  - data/external/*.csv 本地模拟业务数据
  - data/*.txt / *.pdf 知识库资料
  - rag/chroma_db 向量库
  - memory/user_memory.json 轻量记忆

服务与调试层
  - services/chat_service.py 聊天服务入口
  - services/diagnostics_service.py 统一诊断服务
  - evaluations/ 轻量评估
  - scripts/run_project_checks.ps1 一键回归检查
```

核心调用思路是：用户输入问题后，系统先构建轻量上下文，再交给 ReactAgent。Agent 根据提示词判断是否需要调用工具。如果需要，就调用对应工具读取 CSV 或 RAG 资料，最后把工具结果总结成用户能理解的回答。

## 4. Agent 核心能力

### 4.1 工具调用能力

Agent 已注册多种工具，能够根据用户问题选择工具，而不是只靠模型自由回答。

主要工具包括：

| 工具 | 作用 |
| --- | --- |
| `get_model_info` | 查询当前模型、Embedding、知识库和工具能力 |
| `get_agent_diagnostics` | 检查模型、工具、数据文件、向量库和环境变量状态 |
| `get_user_id` | 获取当前会话默认用户 ID |
| `get_user_profile` | 查询用户画像和家庭场景 |
| `get_robot_status` | 查询设备状态、电量、在线状态、异常状态 |
| `get_consumable_status` | 查询主刷、边刷、滤网、拖布、集尘袋寿命 |
| `get_cleaning_history` | 查询清扫历史和清扫效果 |
| `diagnose_fault` | 针对无法回充、漏扫、不出水等问题进行诊断 |
| `rag_summarize` | 从知识库检索并总结答案 |

### 4.2 业务数据查询能力

项目通过本地 CSV 模拟真实业务系统数据。Agent 可以根据用户 ID 查询不同用户的设备、耗材、画像和清扫记录。

例如用户问：

```text
查询用户1004的设备状态
```

Agent 会调用 `get_robot_status(user_id="1004")`，读取 `devices.csv`，返回设备型号、在线状态、电量、当前模式和异常信息。

### 4.3 故障诊断能力

故障诊断工具 `diagnose_fault(issue, user_id="")` 不只是关键词匹配，还会结合：

- 用户描述的问题
- RAG 知识库资料
- 用户设备状态
- 最近清扫历史

输出固定包含：

- 可能原因
- 排查步骤
- 风险等级
- 是否建议售后
- 下一步建议

这样在汇报时可以突出：项目具备“初步售后诊断”的业务价值。

### 4.4 综合分析能力

当用户问“本月表现怎么样”“整体使用情况如何”“有没有需要注意的问题”时，Agent 可以综合调用多个工具：

- 用户画像
- 设备状态
- 耗材状态
- 清扫历史
- RAG 知识库

最后回答会包含使用概览、风险提醒和优化建议。

## 5. 工具调用流程

可以用下面的流程理解 Agent 工具调用：

```text
用户提问
  ↓
Streamlit 或 FastAPI 接收请求
  ↓
ChatService 构建轻量上下文
  ↓
注入 user_id、记忆摘要、规划提示、最近历史
  ↓
ReactAgent 根据 main_prompt 判断任务类型
  ↓
选择并调用工具
  ↓
工具读取 CSV / RAG / 配置状态
  ↓
工具返回可读结果
  ↓
Agent 总结为结构化回答
```

示例：

```text
问题：我的滤网要不要换？
任务类型：耗材类
推荐工具：get_user_id -> get_consumable_status
数据来源：data/external/consumables.csv
回答结构：先给结论，再给关键数据，最后给保养建议
```

## 6. RAG 流程

RAG 是 Retrieval-Augmented Generation，即“检索增强生成”。它的作用是让模型先查本地知识库，再根据资料回答，减少凭空编造。

本项目 RAG 流程：

```text
知识文件
  ↓
rag/vector_store.py 加载 TXT/PDF
  ↓
文本切分
  ↓
Embedding 向量化
  ↓
写入 ChromaDB
  ↓
用户提问
  ↓
rag/rag_service.py 检索相关片段
  ↓
相关性过滤和来源去重
  ↓
模型总结答案
```

当前 RAG 特点：

- 支持 TXT 和 PDF 知识资料。
- 支持来源引用。
- 故障类问题优先检索故障排除资料。
- 如果没有足够资料，会明确回答“知识库中未检索到足够资料”，不强行编造答案。

## 7. 本地模拟业务数据

业务数据位于 `data/external/`。

| 文件 | 说明 |
| --- | --- |
| `users.csv` | 用户画像，包括房屋面积、宠物情况、地毯情况、清扫偏好 |
| `devices.csv` | 设备状态，包括型号、在线状态、电量、模式、异常码 |
| `consumables.csv` | 耗材状态，包括主刷、边刷、滤网、拖布、集尘袋寿命 |
| `cleaning_history.csv` | 清扫历史，包括清扫面积、时长、覆盖率、漏扫次数、异常次数 |
| `records.csv` | 原有报告生成数据，保持原结构 |

为什么先用 CSV？

- 对新手更容易理解。
- 不需要搭建数据库。
- 可以模拟真实业务接口返回的数据。
- 后续可以平滑替换为数据库或真实 API。

## 8. 记忆系统

项目使用轻量记忆系统，不使用数据库。

记忆模块位于 `memory/local_memory.py`，运行时记忆文件为：

```text
memory/user_memory.json
```

记忆只保存非敏感业务信息，例如：

- 用户 ID
- 房屋面积
- 是否有宠物
- 是否有地毯
- 常见故障
- 清扫偏好

不会保存：

- API Key
- 密码
- 电话
- 地址
- 完整隐私聊天内容

记忆读取失败时不会影响正常对话。Streamlit 当前会话记忆优先级高于本地 JSON 记忆。

## 9. 规划系统

项目没有引入复杂多智能体规划框架，而是实现了轻量规划模块：

```text
planning/simple_planner.py
```

它根据用户问题识别任务类型：

- 设备状态类
- 耗材类
- 清扫历史类
- 故障诊断类
- 综合分析类
- 知识问答类

规划模块只做三件事：

- 判断任务类型。
- 给出推荐工具顺序。
- 给出总结重点。

它不会直接执行工具，也不会替代 ReactAgent。这样做的优点是实现简单、可解释、适合演示。

## 10. FastAPI 扩展方向

项目当前已经保留并实现了 FastAPI 服务入口：

```text
api/main.py
```

已有接口包括：

| 接口 | 作用 |
| --- | --- |
| `GET /health` | API 健康检查 |
| `GET /diagnostics` | 返回 Agent 诊断 JSON |
| `POST /chat` | 通过 API 调用 Agent 对话 |
| `GET /users/{user_id}/profile` | 查询用户画像 |
| `GET /users/{user_id}/robot-status` | 查询设备状态 |
| `GET /users/{user_id}/consumables` | 查询耗材状态 |
| `GET /users/{user_id}/cleaning-history` | 查询清扫历史 |

FastAPI 的作用：

- 让项目不只停留在 Web 演示页面。
- 方便后续接入企业系统、客服平台、小程序或 App。
- 让业务系统可以不经过自然语言，直接调用结构化查询接口。

后续可以扩展：

- 用户登录与权限控制。
- 接入真实设备云平台接口。
- 接入数据库。
- 增加售后工单创建接口。
- 增加接口鉴权和访问日志。

## 11. 调试与评估能力

项目提供多种调试方式：

- Streamlit 侧边栏显示模型、Embedding、用户 ID 和上下文摘要。
- `get_agent_diagnostics` 可以检查系统状态。
- `evaluations/run_basic_checks.py` 可以运行轻量评估用例。
- `evaluations/evaluation_report.md` 会生成 Markdown 评估报告。
- `scripts/run_project_checks.ps1` 可以一键检查关键能力。

一键回归检查命令：

```powershell
powershell -ExecutionPolicy Bypass -File scripts/run_project_checks.ps1
```

它会检查：

- Python 关键模块编译
- Agent 自检
- CSV 文件读取
- RAG 无资料兜底
- Streamlit health
- evaluations 基础用例

## 14. 常见答辩问题与答案

### Q1：这个项目和普通聊天机器人有什么区别？

普通聊天机器人主要依赖模型直接回答，而本项目可以调用工具读取业务数据和知识库。例如用户问设备电量时，Agent 会查询 `devices.csv`，而不是凭空猜测。

### Q2：为什么使用 RAG？

RAG 可以让模型先检索本地知识库，再基于资料回答，减少幻觉。对于产品说明、保养方法和故障排查这类知识，RAG 比单纯靠模型记忆更可靠。

### Q3：为什么现在使用 CSV，而不是数据库？

这是二次开发阶段的轻量实现。CSV 更容易理解和演示，可以模拟真实业务数据。后续如果要上线，可以把 CSV 读取层替换为数据库或真实接口。

### Q4：Agent 是怎么知道该调用哪个工具的？

主要依赖 `prompts/main_prompt.txt` 的工具调用规则，以及 `planning/simple_planner.py` 的轻量任务分类。系统会根据关键词和任务类型建议工具顺序，ReactAgent 再完成实际调用。

### Q5：如果用户没有说 user_id 怎么办？

Streamlit 侧边栏有当前用户 ID。用户没有明确指定时，系统会使用侧边栏默认用户；如果用户在问题中说“查询用户1004”，则优先使用用户明确指定的 ID。

### Q6：记忆系统会不会保存隐私？

不会保存 API Key、密码、电话、地址或完整聊天隐私。当前只保存非敏感业务信息，例如是否有宠物、是否有地毯、清扫偏好等。

### Q7：故障诊断是不是完全由模型编造？

不是。故障诊断会优先结合 RAG 知识库资料，如果传入用户 ID，还会参考设备状态和清扫历史。信息不足时会提示用户补充现象。

### Q8：FastAPI 的作用是什么？

Streamlit 适合演示和调试，FastAPI 适合系统集成。后续客服系统、小程序或 App 可以通过 FastAPI 调用 Agent 或业务查询接口。

### Q9：如何证明项目现在是正常的？

可以运行：

```powershell
powershell -ExecutionPolicy Bypass -File scripts/run_project_checks.ps1
```

该脚本会检查关键模块编译、Agent 自检、CSV 读取、RAG 兜底、Streamlit 健康状态和基础评估用例。

### Q10：这个项目后续还能怎么优化？

可以继续优化以下方向：

- 接入真实设备云平台 API。
- 使用数据库替代 CSV。
- 增加用户登录和权限控制。
- 增加售后工单创建能力。
- 增强 RAG 文档质量和召回效果。
- 增加更多自动化评估用例。
- 将记忆系统升级为更稳定的长期记忆。
